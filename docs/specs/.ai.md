# docs/specs/

## 목적
기능 명세·AC(Acceptance Criteria) 문서와 전체 시스템의 기술 개발 명세를 보관한다. 구현 전에 작성하고, 이슈와 1:1로 연결한다.

## 구조

### 기술 개발 명세
- `dev-spec.md` — 전체 시스템의 기술 개발 명세
  - §1 프로젝트 구조 + 기술 스택 (Next.js 15, Supabase, Claude API, Vercel)
  - §2 삭제한 기술 목록 (일론 5단계 원칙 적용 결과)
  - §3 DB 스키마 DDL + RLS 정책 (7개 테이블, Supabase PostgreSQL)
  - §4 API 엔드포인트 + Supabase Realtime 채널 설계
  - §5 구현 단위별 상세 명세 (IU-01~IU-06, 입력/처리/출력/테스트)
  - §6 TDD 테스트 전략 (Vitest + Playwright + MSW + Supabase 로컬)
  - §7 MVP → 고도화 로드맵 (Phase 1~4) + D1~D7 일별 구현 일정

### 개별 기능 명세
- 파일명 형식: `{이슈번호}-{기능명}.md` (예: `0015-user-auth.md`)
- 각 문서에는 배경·목표·AC·비고를 포함한다

## 참조 관계
- `dev-spec.md`는 `docs/whitepaper/project-plan.md` (기획서)의 기술 설계를 구현 수준으로 구체화한 문서다
- 기획서와 dev-spec.md 내용이 상충하는 경우 **dev-spec.md가 구현 기준**이 된다
- 개별 기능 명세는 dev-spec.md의 IU 번호(IU-01~IU-06)를 참조 키로 활용한다

## 규칙
- AC는 구체적이고 검증 가능하게 작성한다
- 구현 완료 후 AC 달성 여부를 문서에 기록한다
- dev-spec.md는 구현 시작 전 확정하며, 변경 시 변경 이유와 날짜를 문서 상단에 기록한다
