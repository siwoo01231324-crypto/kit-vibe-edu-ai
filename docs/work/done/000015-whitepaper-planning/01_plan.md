## RALPLAN-DR Summary

### Principles (핵심 원칙)

1. **근거 기반 기획**: 모든 주장은 docs/background/ 01~11 리서치 또는 추가 조사 출처를 명시한다
2. **페인포인트 직결**: doc 11의 비판적 재검토 결과를 반영하여, 학생 자발적 행동에 의존하지 않는 솔루션 방향을 채택한다
3. **공모전 심사 기준 최적화**: 기술적 완성도 > AI 활용 > 기획력 > 창의성 순서로 심사 기준에 맞춰 기획서를 구성한다
4. **1주 MVP 실현 가능성**: 기획서에 담긴 모든 기능은 04/13까지 실제 구현 가능한 범위여야 한다
5. **단일 문서 체계**: 기획서는 하나의 통합 문서로 작성하되, 섹션별로 독립적으로 읽을 수 있게 구성한다

### Decision Drivers (의사결정 핵심 요인)

1. **마감 시간 (04/13)**: 기획서 작성에 쓸 수 있는 시간이 제한적이므로 효율적 구조가 필수
2. **솔루션 방향 확정 여부**: doc 11에서 "대안 3 + 현재 프레이밍 결합"을 최종 추천했으나, 기획서 작성 시 이 방향을 확정해야 함
3. **심사 가점**: 공모전 가이드라인에서 기획문서 포함 시 가점 명시 — 품질이 직접 점수에 영향

### Viable Options (실행 가능한 옵션)

#### Option A: 단일 통합 기획서 (Recommended)

`docs/whitepaper/project-plan.md` 하나에 모든 섹션을 포함하는 방식.

| 항목 | 평가 |
|------|------|
| 장점 | 읽는 사람(심사위원)이 하나의 흐름으로 파악 가능, 중복 제거 용이, 관리 단순 |
| 장점 | 섹션 간 논리적 연결이 자연스러움 (페인포인트 → 타겟 → 가치제안 → MVP → PRD) |
| 단점 | 문서가 길어질 수 있음 (예상 3,000~5,000줄) |
| 단점 | 병렬 작업이 어려움 (하나의 파일에 여러 에이전트가 동시 작업 불가) |

#### Option B: 섹션별 분리 문서

`docs/whitepaper/01-pain-points.md`, `02-target.md`, ... 형태로 분리.

| 항목 | 평가 |
|------|------|
| 장점 | 병렬 작업 가능, 각 문서가 독립적으로 관리됨 |
| 장점 | 섹션별 출처 관리가 명확 |
| 단점 | 심사위원이 여러 파일을 열어야 함 — 전체 스토리라인 파악이 어려움 |
| 단점 | 섹션 간 중복·불일치 발생 가능, 통합 정리 별도 공수 필요 |

#### Option C: 축소 문서 (10챕터) + MVP 시간 확보

16챕터 대신 핵심 10챕터만 작성하고 절약한 3.5시간을 MVP 구현에 투자.

| 항목 | 평가 |
|------|------|
| 장점 | 공모전 총점 최적화 가능 (기획서 가점 < MVP 본점) |
| 장점 | 심사위원 읽기 부담 감소 |
| 단점 | AC "현업 수준의 프로젝트 기획서" 충족이 어려울 수 있음 |
| 단점 | Lean Canvas, SWOT 등 전략 섹션 축소 시 기획력 점수 하락 가능 |

#### 추천: Option A (단일 통합 기획서)

Option B의 병렬 작업 장점은, 초안을 섹션별로 나눠 작성한 뒤 최종적으로 하나의 문서로 통합하는 방식으로 보완할 수 있다. Option C는 총점 최적화 관점에서 유효하나, 이 이슈(#15)의 범위는 기획서 작성이므로 기획서 품질을 우선한다. 심사위원의 읽기 경험을 우선시한다.

### ADR (Architecture Decision Record)

- **Decision**: 단일 통합 기획서 + 섹션별 초안 병렬 작성 후 통합
- **Drivers**: 심사위원 읽기 경험, 마감 시간 제약, 논리적 일관성
- **Alternatives considered**: (B) 섹션별 분리 문서, (C) 축소 10챕터 + MVP 시간 확보
- **Why B rejected**: 스토리라인 분절, 통합 별도 공수
- **Why C rejected**: 이슈 #15의 범위는 기획서 작성이므로 품질 우선. MVP 시간 배분은 별도 이슈에서 관리
- **Why chosen**: 공모전 제출물로서 하나의 스토리라인이 더 효과적. 병렬 작성은 초안 단계에서 해결
- **Consequences**: 문서가 길어질 수 있으나, 목차와 섹션 구분으로 해결
- **Follow-ups**: 최종 PDF 변환 시 포맷팅 확인 필요
